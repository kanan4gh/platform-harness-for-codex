# Codex ハーネス オンボーディングガイド

このガイドでは、Codex ハーネスの全機能を順番に動作確認しながら、最後に SDD でワンサイクルの開発を体験します。読み終わったとき「全機能が動いていて、1 サイクル回した」状態になることを目標とします。

**前提**:
- GitHub アカウントを持っていること
- VS Code がインストール済みであること（devcontainer を使う場合）
- SDD（スペック駆動開発）の概念は知っていること

---

## 1. このハーネスとは

Codex CLI エージェントを有効に活用するための「文脈注入・プロセス定義」の仕組みです。

### 構成

| 要素 | 場所 | 役割 |
|---|---|---|
| **AGENTS.md** | プロジェクトルート | Codex への常時コンテキスト注入（プロダクト定義・技術スタック・開発プロセス・ワークフロー） |
| **スペックテンプレート** | `.steering/_template/` | 機能実装時にコピーして使うスペックの雛形 |
| **永続ドキュメント** | `docs/` | プロジェクト全体の「何を作るか」「どう作るか」の定義 |
| **devcontainer** | `.devcontainer/` | Codex CLI インストール済みの再現可能な開発環境 |

### 機能チートシート

| 機能 | 呼び出し方 | 用途 |
|---|---|---|
| コンテキスト注入 | 自動（常時読込） | プロダクト定義・技術スタック・開発プロセスを Codex に伝える |
| setup-project | チャットで「setup-project を実行してください」 | プロジェクト初期化（docs/ 作成・AGENTS.md 更新） |
| add-feature | チャットで「add-feature を実行してください」 | 新機能の SDD フローを開始する |

---

## 2. セットアップ

### ステップ 1: テンプレートリポジトリから新しいリポジトリを作成する

GitHub の **"Use this template"** ボタンをクリックして、新しいリポジトリを作成します。

### ステップ 2: 開発環境を準備する

**devcontainer を使う場合（推奨）**:

1. 複製したリポジトリを VS Code で開く
2. 「Reopen in Container」を選択する
3. コンテナのビルドが完了するまで待つ（初回は数分かかります）
4. ターミナルで `codex --version` を実行して Codex CLI が動くことを確認する

**ローカルに直接インストールする場合**:

```bash
npm install -g @openai/codex
codex --version  # インストール確認
```

### ステップ 3: setup-project でプロジェクトを初期化する

Codex CLI のチャットで以下を入力します：

```
setup-project を実行してください
```

Codex が対話形式で以下を行います：

1. プロダクトの概要・目的・ターゲットユーザーをヒアリング
2. `docs/` 配下に 6 ファイルを順番に作成（各ファイルでユーザー確認あり）
3. `AGENTS.md` のプロダクト定義・技術スタック・リポジトリ構造セクションを実際の内容で更新

> 手動で `AGENTS.md` を直接編集したい場合は、付録「手動での AGENTS.md 編集」を参照してください。

---

## 3. 動作確認

`setup-project` が完了したら、ハーネスが正しく動いているか確認します。

### 確認 1: AGENTS.md が読み込まれているか

**チャットに入力**:
```
このプロジェクトの開発プロセスのルールを教えてください
```

**期待される応答**: SDD フロー（Issue → スペック → 実装 → PR）の説明が返ってくる

---

### 確認 2: プロダクト情報が伝わっているか

**チャットに入力**:
```
このプロジェクトのターゲットユーザーを教えてください
```

**期待される応答**: `setup-project` で設定したターゲットユーザーの説明が返ってくる

返ってきた内容が正しくない場合は、`AGENTS.md` のプロダクト定義セクションを修正してください。

---

### 確認 3: add-feature ワークフローを起動する

**チャットに入力**:
```
add-feature を実行してください
```

**期待される応答**: Codex が機能名を尋ねてくる

確認できたら、いったん会話を中断して構いません。

---

以上 3 つの確認がすべて通れば、ハーネスは正しく動作しています。

---

## 4. SDD でワンサイクル回す（総仕上げ）

ハーネスの動作が確認できたら、実際のプロジェクトで最初の機能を SDD で実装します。

### 4-1. GitHub で Issue を作成する

```bash
gh issue create --title "feat: [機能名]" --body "[機能の概要]"
```

### 4-2. add-feature ワークフローでスペック作成を開始する

**チャットに入力**:
```
add-feature を実行してください
```

Codex が機能名を尋ねてきたら答えます。その後、以下の順で進めます。

1. **requirements.md を記入する** — Codex が下書きを提案します。内容を確認・修正して承認します
2. **design.md を記入する** — 実装アプローチを確認・修正して承認します
3. **tasklist.md を記入する** — タスクをフェーズ分けして確認・修正して承認します

> 各ステップで Codex が確認を求めてきます。必ず内容を確認してから承認してください。

### 4-3. 実装する

tasklist.md を承認したら実装フェーズに入ります。Codex がタスクを 1 つずつ実行します。

実装後の動作確認はターミナルで行います（技術スタックに応じてコマンドを変更してください）：

```bash
# 例（Python/uv の場合）
uv run pytest           # テストが通ることを確認
uv run ruff check .     # Lint エラーがないことを確認
```

### 4-4. PR を作成してマージする

```bash
gh pr create
```

または Codex に依頼します：
```
PR を作成してください
```

Codex が PR タイトルと説明文を自動生成します。内容を確認してマージします。

## 付録: 手動での AGENTS.md 編集

`setup-project` を使わずに手動で `AGENTS.md` を編集することもできます。

### プロダクト定義セクション

`AGENTS.md` を開き、以下のプレースホルダーを書き換えます：

```markdown
## プロダクト定義

### プロダクト名
[プロダクト名とサブタイトルを記入]  ← ここを書き換える

### ビジョン
[このプロダクトが目指す世界観を 1〜3 文で記入]  ← ここを書き換える
```

### 技術スタックセクション

```markdown
## 技術スタック

### 使用技術
[言語・フレームワーク・主要ライブラリ・ツールを記入]  ← ここを書き換える
```

> **「開発プロセス原則」と「ワークフロー」セクションは編集不要**です。SDD の開発プロセスルールが定義されており、ハーネス共通で使います。

## 付録: devcontainer のカスタマイズ

`.devcontainer/devcontainer.json` を編集して、プロジェクトの技術スタックに合わせた機能を追加できます。

```jsonc
{
  "features": {
    "ghcr.io/devcontainers/features/github-cli:1": {},
    // AWS CLI を使う場合は以下のコメントを外す
    // "ghcr.io/devcontainers/features/aws-cli:1": {}
  }
}
```

`.devcontainer/postCreate.sh` を編集して、プロジェクト固有のセットアップを追加できます。

```bash
# Python プロジェクトの場合
curl -LsSf https://astral.sh/uv/install.sh | sh
export PATH="$HOME/.local/bin:$PATH"
uv sync
```
