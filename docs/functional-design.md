# 機能設計書 (Functional Design Document)

## 設計指針

このプロダクトは、画像生成プロンプトを複数のプロンプト部品として扱い、それらを合成して最終的な画像生成プロンプトを作る。MVP 0〜2はCodex CLIまたはCodex SDKを使ったローカルツールとして実装する。MVP 0ではコーデ抽出の仮説検証に集中し、段階的に人物固定、ネガティブ抑制、背景、撮影、ポーズ、振り返り、コラージュへ拡張する。

---

### 指針1: プロンプト部品を中核モデルにする

**方針**:
画像生成プロンプトを一枚岩のテキストではなく、独立したプロンプト部品の組み合わせとして管理する。

**根拠**:
人物固定、コーディネート、背景、撮影、ポーズ、ネガティブ抑制は改善タイミングも評価観点も異なる。部品として分けることで、どの要素が良かったか、悪かったかを振り返りやすくする。

**影響範囲**:
- プロンプト部品ごとに保存、編集、評価、再利用の単位を持つ
- 最終プロンプトは部品合成の成果物として扱う
- フィードバックは最終プロンプトだけでなく、原因となる部品へ紐づけられるようにする

---

### 指針2: MVP 0はコーデ抽出品質の検証に絞る

**方針**:
MVP 0では、コーデ画像からコーデ抽出プロンプトを作成し、ChatGPTで同じ画像から抽出した結果と比較する。

**根拠**:
最初に検証すべき仮説は「本プロダクトのコーデ抽出結果がChatGPTと同等に有用か」である。人物固定や背景指定などを混ぜると、抽出品質の評価が曖昧になる。

**影響範囲**:
- MVP 0では固定人物、ネガティブ抑制、最終プロンプト合成を実装しない
- 比較評価では、良かった/悪かったの主観評価を必ず記録できるようにする
- ChatGPT結果はユーザーが貼り付ける入力として扱う

---

### 指針3: APIコストがかかる画像生成は後回しにする

**方針**:
MVP 0〜2ではCodex CLIまたはCodex SDKを使い、画像生成APIを直接呼び出さない。ChatGPT Web UIを使うのは最終画像生成時に限定し、ローカルツールはChatGPTへ貼り付けられる最終プロンプトを出力する。

**根拠**:
初期段階では、生成APIコストをかけるよりも、抽出・合成・評価のワークフローを検証することが重要である。

**影響範囲**:
- アプリ内で生成画像を作らない
- 元画像と最終プロンプトを外部UIへ持ち出しやすい形で表示する
- 生成結果は、ユーザーが必要に応じてアップロードまたは記録する

---

### 指針5: Codexをローカル実行基盤にする

**方針**:
MVP 0〜2のLLM処理は、Codex CLIまたはCodex SDKから実行する。OpenAI APIキーを前提にしたサーバー型アプリにはしない。

**根拠**:
ユーザーはChatGPT/Codex認証での利用を重視しており、APIキー課金を前提にしたWebバックエンドは初期方針に合わない。Codex CLI/SDKなら、ローカルファイル、画像入力、履歴ファイル、プロンプト生成を同じ作業環境で扱える。

**影響範囲**:
- MVP 0〜2はローカルCLIまたはローカルSDKラッパーとして提供する
- プロンプト部品と評価履歴はローカルファイルまたは軽量DBに保存する
- 将来Web UI化する場合も、Codex実行部分はアダプタとして分離する

---

### 指針4: 失敗再生成と良い派生生成を区別する

**方針**:
同じコーディネートでやり直した試行を、失敗による再生成、気に入った画像からの派生生成、比較検討のためのバリエーション生成に分類する。

**根拠**:
やり直し回数は少ないほど良いが、良い画像を起点に背景・撮影・ポーズを変える試行はプロダクト価値であり、失敗として扱うべきではない。

**影響範囲**:
- 振り返りでは「良かった/悪かった」と試行種別を分けて記録する
- 失敗再生成回数は改善対象として扱う
- 派生生成は成功パターンの展開として扱う

## コンポーネント設計

### Image Input

**責務**:
- 実写画像、アニメ風イラスト画像、人物元画像、生成済み画像を受け取る
- 入力画像の用途を区別する

**インターフェース**:
- 画像アップロード
- 入力種別選択: コーデ画像、人物元画像、生成済み画像
- 画像種別選択: 実写、アニメ風イラスト、不明

**実装の要点**:
- MVP 0ではコーデ画像の入力だけを必須とする
- 画像種別はユーザー指定を優先し、将来的に推定を追加できる余地を残す
- 破損画像、未対応形式、サイズ超過は抽出処理前に止める

---

### Prompt Part Store

**責務**:
- プロンプト部品を保存、編集、再利用する
- 部品ごとの出自、導入MVP、評価履歴を保持する

**扱う部品**:
- コーデ抽出プロンプト
- 人物固定プロンプト
- ネガティブ抑制プロンプト
- 背景プロンプト
- 撮影プロンプト
- ポーズプロンプト

**インターフェース**:
- 部品作成
- 部品編集
- 部品選択
- 部品保存
- 部品評価の参照

**実装の要点**:
- 部品には、ユーザー入力、画像抽出、プリセット、LLM改善提案などの出自を持たせる
- MVP 0ではコーデ抽出プロンプトだけを保存対象とする
- MVP 1では人物固定とネガティブ抑制をユーザー入力のみで扱う

---

### Coordinate Extractor

**責務**:
- コーデ画像から服装、小物、色、素材、シルエット、スタイリングを抽出する
- 抽出結果を画像生成に貼り付けやすいコーデ抽出プロンプトとして出力する

**インターフェース**:
- 入力: コーデ画像、画像種別
- 出力: コーデ抽出プロンプト、抽出メタ情報

**実装の要点**:
- MVP 0の最重要コンポーネントとする
- アニメ風イラスト画像では、画風ではなく服装構成を抽出する
- 不確実な要素は断定せず、曖昧さが残る表現にする

---

### ChatGPT Comparison

**責務**:
- 本プロダクトの抽出結果とChatGPTで得た抽出結果を比較する
- ユーザーの主観評価を記録する

**インターフェース**:
- ChatGPT抽出結果の貼り付け
- 並列表示
- 良かった/悪かった評価
- 理由カテゴリと自由記述

**実装の要点**:
- MVP 0に含める
- 比較観点は、服装要素の網羅性、色・柄、素材感、シルエット、プロンプトとしての使いやすさとする
- 評価は自動採点ではなく、ユーザーの主観を一次情報として保存する

---

### Prompt Composer

**責務**:
- 選択されたプロンプト部品を結合し、最終的な画像生成プロンプトを作る
- ポジティブ側とネガティブ側のプロンプトを整理する

**インターフェース**:
- 入力: コーデ抽出、人物固定、背景、撮影、ポーズ、ネガティブ抑制
- 出力: 最終プロンプト、貼り付け用テキスト

**実装の要点**:
- MVP 1から導入する
- MVP 1では人物固定とネガティブ抑制をユーザー入力のまま合成する
- 合成後の最終プロンプトはユーザーが編集できるようにする

---

### Prompt Improvement Advisor

**責務**:
- 人物固定プロンプトとネガティブ抑制プロンプトの改善案を提示する
- 生成結果フィードバックをもとに改善候補を作る

**インターフェース**:
- 既存部品に対する改善提案
- 採用、編集、破棄
- 今回だけ適用、保存済み部品へ反映

**実装の要点**:
- MVP 2から導入する
- 提案は自動適用しない
- ユーザーが採用した案だけを最終プロンプトまたは保存済み部品に反映する

---

### Variation Selector

**責務**:
- 背景、撮影、ポーズの抽出・選択・編集を扱う
- 気に入った画像からの派生生成を支援する

**インターフェース**:
- 背景プロンプト抽出
- 撮影プロンプト抽出
- ポーズプロンプト抽出
- アングル候補選択
- 撮影スタイル候補選択
- 背景候補選択

**実装の要点**:
- MVP 3から導入する
- コーデ画像由来、人物元画像由来、プリセット、ユーザー編集を出自として区別する
- 失敗再生成ではなく派生生成として記録できる導線を用意する

---

### Feedback Tracker

**責務**:
- 良かった/悪かったの主観評価を保存する
- 失敗による再生成回数と、良い派生生成を区別して記録する

**インターフェース**:
- 良かった/悪かった選択
- 理由カテゴリ選択
- 自由記述
- 試行種別選択
- 紐づけ対象選択

**実装の要点**:
- MVP 0ではChatGPT比較評価に紐づける
- MVP 2以降では生成結果フィードバックにも使う
- フィードバック対象は、入力画像、プロンプト部品、最終プロンプト、生成済み画像を想定する

---

### Layout Template Composer

**責務**:
- 複数の生成済み画像をテンプレートに沿って一枚の画像へ配置する
- 雑誌風コラージュやコーデ説明画像を作る

**インターフェース**:
- コーデレイアウトテンプレート読み込み
- 画像スロット割り当て
- テキストスロット編集
- 一枚画像として出力

**実装の要点**:
- Post-MVPで導入する
- 画像生成プロンプトとは別のレイアウト定義として扱う
- 全身、正面、後ろ、横などの視点別画像を配置できるようにする

## ユースケースフロー

### MVP 0: コーデ抽出とChatGPT比較

```
1. ユーザーがコーデ画像をアップロードする
2. ユーザーが画像種別を指定する
3. Coordinate Extractor がコーデ抽出プロンプトを生成する
4. ユーザーが同じ画像をChatGPTで抽出した結果を貼り付ける
5. ChatGPT Comparison が両方の結果を並べて表示する
6. ユーザーが良かった/悪かったを記録する
7. 評価理由と自由記述を保存する
```

### MVP 1: ユーザー入力部品による最終プロンプト作成

```
1. ユーザーがコーデ抽出プロンプトを用意する
2. ユーザーが人物固定プロンプトを選択または入力する
3. ユーザーがネガティブ抑制プロンプトを選択または入力する
4. Prompt Composer が最終プロンプトを生成する
5. ユーザーが内容を確認・編集する
6. ユーザーがChatGPT貼り付け用テキストをコピーする
```

### MVP 2: LLM提案による部品改善

```
1. ユーザーが既存の人物固定プロンプトまたはネガティブ抑制プロンプトを選ぶ
2. Prompt Improvement Advisor が改善案を提示する
3. ユーザーが採用、編集、破棄を選ぶ
4. 採用された改善案だけを今回の最終プロンプトまたは保存済み部品へ反映する
5. 生成結果の良かった/悪かったを Feedback Tracker に記録する
```

### MVP 3: 背景・撮影・ポーズの派生生成

```
1. ユーザーが気に入った画像またはコーデ画像を選ぶ
2. Variation Selector が背景、撮影、ポーズの部品化を支援する
3. ユーザーが背景、撮影、ポーズ、アングル候補を選ぶ
4. Prompt Composer が派生生成用の最終プロンプトを作る
5. ユーザーが試行種別を「気に入った画像からの派生生成」として記録する
```

### Post-MVP: コラージュ・コーデ説明画像

```
1. ユーザーが生成済み画像を複数選択する
2. ユーザーがコーデレイアウトテンプレートを選ぶ
3. Layout Template Composer が画像スロットとテキストスロットを表示する
4. ユーザーが画像と説明文を配置する
5. 一枚の雑誌風コラージュまたはコーデ説明画像として出力する
```

## エラー処理方針

- 入力画像が未対応形式、破損、サイズ超過の場合は抽出処理を実行せず、原因を表示する
- コーデ抽出に失敗した場合は、再試行、画像差し替え、手動入力の選択肢を提示する
- ChatGPT比較用テキストが未入力の場合は、比較評価を未完了として保存できる
- プロンプト部品が不足している場合は、どの部品が不足しているかを表示する
- MVP 1では人物固定プロンプトやネガティブ抑制プロンプトの自動補完を行わない
- LLM改善提案に失敗した場合は、既存の部品を維持し、ユーザー入力で続行できるようにする
- フィードバック保存に失敗した場合は、評価内容を失わないように再送または一時保持できる設計にする
- 画像生成APIはMVP 0〜3では呼び出さないため、生成API失敗はこの段階のエラー対象に含めない
