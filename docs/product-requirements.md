# プロダクト要求定義書 (Product Requirements Document)

## プロダクト概要

### 名称

Coord Prompt Studio（仮称）

### プロダクトコンセプト

参照画像または既存のコーデ再現プロンプトからコーディネート要素を抽出し、人物固定、コーディネート、背景、撮影、ネガティブ抑制などのプロンプト部品を組み合わせて、ChatGPTなどの画像生成画面に貼り付けやすい最終的な画像生成プロンプトを作成するツール。

### プロダクトビジョン

画像生成における「服装の再現」「人物設定の固定」「不要要素の抑制」を一連のワークフローとして扱い、毎回の手作業プロンプト調整を減らす。MVP 0〜2ではCodex CLIまたはCodex SDKを使ったローカルツールとして実装し、ユーザーのChatGPT/Codex認証を利用する。最終画像生成では、ユーザーが元画像と最終プロンプトをChatGPT Web UIへ貼り付けて生成する。

最初の検証段階であるMVP 0では、コーディネート画像からコーデ再現プロンプトを抽出する機能だけに絞る。ChatGPTで同じ画像から抽出した結果と比較し、本プロダクトの抽出結果がChatGPTと同等の品質に達するという仮説を検証する。ChatGPT Web UIを使うのは、原則として最終画像生成時に限定する。

### 目的

- 実写画像、アニメ風イラスト画像、既存のコーデ再現プロンプトから、服装・小物・色・素材感・スタイリングを画像生成向けのプロンプトへ変換する
- MVP 0では、コーディネート画像からコーデ再現プロンプトを抽出する仮説検証に集中する
- MVP 0の抽出結果をChatGPTで同様に抽出した結果と比較し、品質差を評価する
- あらかじめ設定した人物固定プロンプトを合成し、同一キャラクター・同一人物設定の一貫性を保つ
- ネガティブ抑制プロンプトを合成し、品質低下・不要な表現・崩れを抑える
- 人物固定プロンプトとネガティブ抑制プロンプトは、まずユーザーが与えた内容をそのまま使う段階から始め、後続MVPでLLMによる改善提案を追加する
- アニメ風イラスト画像を入力にした場合でも、実写としてコーディネートを再現するための指示を自動で加える
- 元画像と最終プロンプトを、ユーザーがChatGPTなどの画像生成UIへ貼り付けやすい形で出力する
- 外部の画像生成UIで得た結果に対するユーザーフィードバックをもとに、次回のプロンプト改善へつなげる
- ユーザーが思いつきにくいカメラアングルや人物ポーズの変更候補を提示し、再生成用プロンプトを作りやすくする
- 撮影スタイルと背景について、コーデ画像のものを使う、人物元画像のものを使う、候補から自由に選ぶ、という選択肢を段階的に提供する
- MVP 0〜2ではOpenAI APIキーを前提にせず、Codex CLIまたはCodex SDK経由でローカルに処理する
- 最終画像生成はChatGPT Web UIで行い、アプリ内の画像生成API呼び出しは行わない

## コアモデル

このプロダクトが行うことは、画像生成プロンプトを複数のプロンプト部品の組み合わせとして構築することである。各部品は独立して作成・保存・編集でき、最終的に1つの画像生成プロンプトへ合成される。

### プロンプト部品

| 部品 | 役割 | 主な入力元 | 導入段階 |
|------|------|------------|----------|
| コーデ抽出プロンプト | 服装、小物、色、素材、シルエット、スタイリングを表す | コーデ画像、既存のコーデ再現プロンプト | MVP 0 |
| 人物固定プロンプト | 同一人物・同一キャラクターとして生成するための外見・雰囲気を表す | ユーザー入力、MVP 2以降はLLM改善提案 | MVP 1 |
| ネガティブ抑制プロンプト | 崩れ、不要要素、アニメ化、品質低下などを抑える | ユーザー入力、MVP 2以降はLLM改善提案 | MVP 1 |
| 背景プロンプト | 背景をコーデ画像由来、人物元画像由来、指定なし、候補選択のいずれかで表す | コーデ画像からの抽出、人物元画像からの抽出、プリセット、ユーザー編集 | MVP 3 |
| 撮影プロンプト | 撮影スタイル、カメラアングル、視線、距離感、ライティング、写真の質感を表す | コーデ画像からの抽出、人物元画像からの抽出、プリセット、ユーザー編集、将来的なLLM提案 | MVP 3 |
| ポーズプロンプト | 人物の姿勢、動き、手足の配置、服の見え方を表す | コーデ画像からの抽出、人物元画像からの抽出、プリセット、ユーザー編集、将来的なLLM提案 | MVP 3 |

### 合成方針

- 最終的な画像生成プロンプトは、複数のプロンプト部品を順序立てて結合した成果物とする
- MVP 0では、コーデ抽出プロンプト単体の品質検証に集中する
- MVP 1では、ユーザーが与えた人物固定プロンプトとネガティブ抑制プロンプトをそのまま合成する
- MVP 2では、人物固定プロンプトとネガティブ抑制プロンプトに対してLLMが改善案を提案する
- MVP 3では、背景プロンプト、撮影プロンプト、ポーズプロンプトを抽出・選択・編集できる部品として追加する
- 各部品は最終プロンプトへ自動合成されるが、合成後のプロンプトはユーザーが確認・編集できる

## ターゲットユーザー

### プライマリーペルソナ

- 役割: 画像生成を使ってキャラクター画像、ファッション画像、SNS向けビジュアル、創作素材を作る個人クリエイター
- 課題: 参照画像のコーディネートを言語化する手間が大きく、人物設定や品質抑制プロンプトを毎回組み直す必要がある
- 期待: 画像または既存プロンプトを渡すだけでコーディネート要素が整理され、固定人物設定を保った最終プロンプトが得られる

### セカンダリーペルソナ

- 役割: 複数パターンの衣装案・スタイリング案を効率よく試したいデザイナー、企画者、プロンプト作成者
- 課題: 衣装差分の作成時に、人物設定や生成品質を安定させながら試行するのが難しい
- 期待: コーディネート抽出結果と最終プロンプトを確認・調整しながら、外部の画像生成UIで複数の生成案を作成できる

## 成功指標 (KPI)

このプロダクトでは、画像生成結果の良し悪しは最終的にユーザーの主観で明確に判定される。そのため、KPIは自動スコアよりも、振り返りの過程でユーザーが「良かった」「悪かった」を記録できることを中心に置く。

### プライマリー KPI

- 振り返り記録率: プロンプト作成や生成試行のあと、ユーザーが「良かった」「悪かった」を記録できる
- 主観評価の明確性: 評価はまず「良かった」「悪かった」の二択で記録できる
- 評価理由の記録: 「悪かった」または「良かった」の理由を、カテゴリ選択と自由記述で残せる
- 試行履歴との紐づけ: 評価は、入力画像、プロンプト部品、最終プロンプト、ChatGPT比較結果、生成済み画像のいずれかに紐づけて保存できる
- やり直し回数: 同じコーディネートで「悪かった」ために画像生成をやり直した回数を記録できる。少ないほど良い状態とみなす
- 派生生成との区別: 気に入った画像を起点に背景、撮影、ポーズを変える試行は、失敗によるやり直しではなく良い派生生成として記録できる

### セカンダリー KPI

- MVP 0では、本プロダクトのコーデ抽出結果とChatGPTの抽出結果のどちらが良かったかをユーザーが記録できる
- MVP 1以降では、最終プロンプトや生成結果が良かったか悪かったかをユーザーが記録できる
- 「悪かった」理由として、人物が固定できていない、コーデ再現が弱い、ポーズが違う、背景が違う、撮影スタイルが違う、アニメ調が残っている、品質が低い、を記録できる
- 「良かった」理由として、人物が安定している、コーデ再現が良い、背景が合っている、撮影スタイルが良い、ポーズが良い、全体の雰囲気が良い、を記録できる
- 同じコーディネートに対する試行を、失敗による再生成、気に入った画像からの派生生成、比較検討のためのバリエーション生成に分類できる
- 蓄積した主観評価を、プロンプト部品や最終プロンプト生成ルーチンの改善判断に使える

## MVP段階定義

| 段階 | 目的 | 含める機能 | 含めない機能 |
|------|------|------------|--------------|
| MVP 0 | コーデ画像からChatGPT同等のコーデ再現プロンプトを抽出できるか検証する | 画像入力、コーデ抽出、ChatGPT比較評価 | 人物固定、ネガティブ抑制、最終プロンプト合成 |
| MVP 1 | ユーザーが与えたプロンプトを合成し、ChatGPT貼り付け用の最終プロンプトを作る | 固定人物プロンプト管理、ネガティブ抑制プロンプト管理、最終プロンプト合成、貼り付け用出力 | LLMによる人物固定/ネガティブ抑制の提案・改善 |
| MVP 2 | LLMと一緒にプロンプトを改善する | 人物固定プロンプト改善提案、ネガティブ抑制プロンプト改善提案、生成結果フィードバック | 撮影スタイル、背景、アングル、ポーズの体系的バリエーション |
| MVP 3 | 構図・撮影表現のバリエーションを増やす | 背景/撮影/ポーズプロンプト抽出、アングル候補、ポーズ候補、撮影スタイル候補、背景指定候補 | 画像生成API連携、生成画像の自動評価 |

### 実装形態

- MVP 0〜2は、Codex CLIまたはCodex SDKを使ったローカルツールとして実装する
- ローカルツールは、画像入力、プロンプト部品管理、最終プロンプト生成、振り返り記録を扱う
- Codexの認証はユーザーのChatGPT/Codexログインを利用する方針とし、OpenAI APIキー課金を前提にしない
- ChatGPT Web UIを使うのは、原則として最終画像生成時のみとする

## 機能要件

### コア機能 (MVP 0)

MVP 0は、以降の固定人物プロンプト合成やネガティブ抑制プロンプト合成へ進む前の仮説検証段階とする。最初に検証する仮説は「コーディネート画像から、ChatGPTと同等のコーデ再現プロンプトを抽出できる」である。

#### コーディネート入力

**ユーザーストーリー**:
ユーザーとして、コーディネートの元になる画像を入力したい。なぜなら、その画像からコーデ再現プロンプトを抽出し、ChatGPTの抽出結果と比較したいから。

**受け入れ条件**:
- [ ] ユーザーは実写画像をアップロードできる
- [ ] ユーザーはアニメ風イラスト画像をアップロードできる
- [ ] 入力種別をユーザーが指定、またはシステムが推定できる
- [ ] 入力内容はコーディネート抽出処理に渡される
- [ ] 対応していない形式や破損画像の場合、分かりやすいエラーが表示される

**優先度**: P0（必須）

---

#### コーディネート抽出

**ユーザーストーリー**:
ユーザーとして、入力された画像から服装・小物・色・素材・雰囲気を抽出したい。なぜなら、画像生成向けに安定したコーデ再現プロンプトを作りたいから。

**受け入れ条件**:
- [ ] 画像からトップス、ボトムス、靴、バッグ、アクセサリーなどの主要要素を抽出できる
- [ ] 色、柄、素材感、シルエット、季節感、スタイリングの特徴を抽出できる
- [ ] 抽出結果は、画像生成に貼り付けやすいコーデ再現プロンプトとして表示される
- [ ] 不確実な要素は断定せず、曖昧さが分かる表現にできる
- [ ] アニメ風イラスト画像を入力にした場合、見た目の画風ではなく、服装構成・色・素材・シルエットを抽出できる

**優先度**: P0（必須）

---

#### ChatGPT比較評価

**ユーザーストーリー**:
ユーザーとして、同じコーデ画像をChatGPTで抽出した結果と本プロダクトの抽出結果を比較したい。なぜなら、本プロダクトの抽出品質がChatGPTと同等かを最初に検証したいから。

**受け入れ条件**:
- [ ] ユーザーはChatGPTで得たコーデ抽出結果を貼り付けられる
- [ ] 本プロダクトの抽出結果とChatGPTの抽出結果を同じ画面で比較できる
- [ ] 比較観点として「服装要素の網羅性」「色・柄の正確さ」「素材感の表現」「シルエットの表現」「画像生成プロンプトとしての使いやすさ」を記録できる
- [ ] 比較結果として、同等・不足・過剰・要改善の評価を残せる
- [ ] 評価結果は入力画像と抽出プロンプトに紐づけて保存できる

**優先度**: P0（必須）

### コア機能 (MVP 1)

MVP 1では、MVP 0で抽出品質の仮説を検証した後、ユーザーが与えた固定人物プロンプトとネガティブ抑制プロンプトをそのまま使って、ChatGPT貼り付け用の最終プロンプト生成へ拡張する。この段階では、LLMが人物固定プロンプトやネガティブ抑制プロンプトの内容を提案・改善しない。

#### テキスト入力によるコーデ再現プロンプト取り込み

**ユーザーストーリー**:
ユーザーとして、既存のコーデ再現プロンプトを入力したい。なぜなら、画像ではなく作成済みプロンプトを起点に最終プロンプトを組み立てたいから。

**受け入れ条件**:
- [ ] ユーザーは既存のコーデ再現プロンプトをテキスト入力できる
- [ ] 既存のコーデ再現プロンプトから服装要素を抽出・整理できる
- [ ] 抽出・整理結果を固定人物プロンプトやネガティブ抑制プロンプトと合成できる

**優先度**: P1（重要）

---

#### 固定人物プロンプト管理

**ユーザーストーリー**:
ユーザーとして、自分で用意した人物固定プロンプトを保存して使いたい。なぜなら、衣装だけを変えながら同じ人物設定で画像を生成したいから。

**受け入れ条件**:
- [ ] ユーザーは固定人物プロンプトを登録・編集・削除できる
- [ ] 登録済みの固定人物プロンプトを画像生成時に選択できる
- [ ] 選択した固定人物プロンプトはコーディネート抽出プロンプトと合成される
- [ ] MVP 1では、システムは固定人物プロンプトの内容を自動提案・自動改善しない
- [ ] MVP 1では、ユーザーが入力した固定人物プロンプトを原則としてそのまま最終プロンプトに反映する

**優先度**: P1（重要）

---

#### ネガティブ抑制プロンプト管理

**ユーザーストーリー**:
ユーザーとして、自分で用意したネガティブ抑制プロンプトを保存して使いたい。なぜなら、生成結果の崩れや意図しない要素、とくにアニメ風入力から実写生成へ変換する際のアニメ化を減らしたいから。

**受け入れ条件**:
- [ ] ユーザーはネガティブ抑制プロンプトを登録・編集・削除できる
- [ ] 登録済みのネガティブ抑制プロンプトを画像生成時に選択できる
- [ ] 選択したネガティブ抑制プロンプトは最終プロンプトに反映される
- [ ] アニメ風イラスト画像を入力にした場合、アニメ調・イラスト調・非実写表現を抑制するネガティブ指示を追加できる
- [ ] MVP 1では、システムはネガティブ抑制プロンプトの内容を自動提案・自動改善しない
- [ ] MVP 1では、ユーザーが入力したネガティブ抑制プロンプトを原則としてそのまま最終プロンプトに反映する

**優先度**: P1（重要）

---

#### 画像生成プロンプト組み立て

**ユーザーストーリー**:
ユーザーとして、抽出されたコーディネート、固定人物プロンプト、ネガティブ抑制プロンプトを組み合わせた完成プロンプトを得たい。なぜなら、生成前に内容を確認・調整したいから。

**受け入れ条件**:
- [ ] コーディネート抽出結果と固定人物プロンプトからポジティブプロンプトを生成できる
- [ ] ネガティブ抑制プロンプトを別枠で保持できる
- [ ] アニメ風イラスト画像を入力にした場合、「実写としてコーディネートを再現する」趣旨の指示を追加できる
- [ ] 生成前に完成プロンプトを確認できる
- [ ] ユーザーは完成プロンプトを手動編集できる

**優先度**: P1（重要）

---

#### ChatGPT貼り付け用出力

**ユーザーストーリー**:
ユーザーとして、元画像と完成した画像生成プロンプトをChatGPTへ貼り付けやすい形で受け取りたい。なぜなら、MVP 1ではAPIコストをかけずに外部の画像生成UIで生成したいから。

**受け入れ条件**:
- [ ] 最終プロンプトをコピーできる
- [ ] 元画像を一緒に使う必要があることをユーザーが理解できる表示にする
- [ ] ChatGPTへ貼り付けるための手順文または貼り付け用テキストを生成できる
- [ ] MVPでは画像生成APIを呼び出さない
- [ ] プロンプト作成に使った入力と最終プロンプトの関連を履歴として確認できる

**優先度**: P1（重要）

---

### コア機能 (MVP 2)

MVP 2では、MVP 1でユーザー入力のみだった人物固定プロンプトとネガティブ抑制プロンプトに対し、LLMが改善候補を提案する。提案は自動適用せず、ユーザーが採用したものだけを保存・反映する。

#### 人物固定プロンプト改善提案

**ユーザーストーリー**:
ユーザーとして、コーデ画像や抽出プロンプトに合わせて人物固定プロンプトの改善案を提案してほしい。なぜなら、自分で用意した人物固定プロンプトが、衣装や生成意図に対して十分か判断しにくいから。

**受け入れ条件**:
- [ ] ユーザーは既存の人物固定プロンプトに対する改善提案を生成できる
- [ ] 改善提案は、人物の一貫性を保つために不足している外見特徴、年齢層、髪型、体型、雰囲気、写真品質の記述を補える
- [ ] コーデ画像の服装要素と衝突する人物固定要素がある場合、注意点として表示できる
- [ ] 提案は元の人物固定プロンプトとは別に表示され、ユーザーが採用・編集・破棄を選べる
- [ ] 採用された改善案だけが保存済みプロンプトまたは今回の最終プロンプトに反映される

**優先度**: P1（重要）

---

#### ネガティブ抑制プロンプト改善提案

**ユーザーストーリー**:
ユーザーとして、入力画像や生成目的に合わせたネガティブ抑制プロンプトの改善案を提案してほしい。なぜなら、アニメ化、人物崩れ、背景のズレ、服装再現の弱さなどを事前に抑えたいから。

**受け入れ条件**:
- [ ] ユーザーは既存のネガティブ抑制プロンプトに対する改善提案を生成できる
- [ ] アニメ風イラスト画像を実写化する場合、アニメ調・イラスト調・2D感を抑える提案を含められる
- [ ] 人物固定が弱くなりそうな場合、顔・髪型・体型・年齢感のブレを抑える提案を含められる
- [ ] 背景や撮影スタイルを固定したい場合、不要な背景変化や過剰演出を抑える提案を含められる
- [ ] 提案は自動適用せず、ユーザーが採用・編集・破棄を選べる

**優先度**: P1（重要）

---

#### 生成結果フィードバック

**ユーザーストーリー**:
ユーザーとして、ChatGPTなどで生成した結果の問題点を記録したい。なぜなら、その問題をもとに人物固定プロンプト、ネガティブ抑制プロンプト、最終プロンプト生成ルーチンを改善したいから。

**受け入れ条件**:
- [ ] ユーザーは生成結果に対して問題カテゴリを選択できる
- [ ] 問題カテゴリには「人物が固定できていない」「ポーズの再現が弱い」「背景がイメージと違う」「コーディネートの再現が弱い」「アニメ調が残っている」「品質が低い」を含める
- [ ] ユーザーは試行の種別を「失敗による再生成」「気に入った画像からの派生生成」「比較検討のためのバリエーション生成」から選択できる
- [ ] 同じコーディネートで失敗による再生成が何回発生したかを記録できる
- [ ] ユーザーは問題点を自由記述で補足できる
- [ ] フィードバックは、どの入力・固定人物プロンプト・ネガティブ抑制プロンプト・最終プロンプトに対するものか紐づけて保存できる
- [ ] フィードバックをもとに、次回用の改善プロンプト案を生成できる

**優先度**: P1（重要）

---

### コア機能 (MVP 3)

MVP 3では、最終プロンプトのバリエーション生成を扱う。コーデ画像や人物元画像から背景・撮影・ポーズ表現を部品化できるようにし、さらにカメラアングル、人物ポーズ、撮影スタイル、背景の扱いを選択式で提供する。

#### 背景・撮影・ポーズプロンプト抽出

**ユーザーストーリー**:
ユーザーとして、コーデ画像の背景、撮影の雰囲気、人物ポーズが気に入った場合、それを背景プロンプト、撮影プロンプト、ポーズプロンプトとして部品化したい。なぜなら、服装だけでなく、写真全体の見え方や人物の見せ方も再利用したいから。

**受け入れ条件**:
- [ ] ユーザーはコーデ画像から背景要素を抽出できる
- [ ] 背景抽出では、場所、空間、時間帯、季節感、照明、背景の混み具合を表現できる
- [ ] ユーザーはコーデ画像から撮影要素を抽出できる
- [ ] 撮影抽出では、カメラ距離、アングル、構図、視線、ライティング、写真の質感を表現できる
- [ ] ユーザーはコーデ画像からポーズ要素を抽出できる
- [ ] ポーズ抽出では、立ち方、座り方、歩き、振り返り、手の位置、服のシルエットが見える姿勢を表現できる
- [ ] 抽出された背景プロンプト、撮影プロンプト、ポーズプロンプトは、それぞれ独立したプロンプト部品として保存・編集できる
- [ ] 抽出結果は、コーデ抽出プロンプトとは分けて表示される
- [ ] ユーザーは抽出された部品を今回だけ使うか、再利用可能な部品として保存するか選べる

**優先度**: P1（重要）

---

#### アングル・ポーズ変更候補

**ユーザーストーリー**:
ユーザーとして、カメラアングルや人物ポーズを変えた再生成案を提案してほしい。なぜなら、自分では思いつきにくい構図のバリエーションを試したいから。

**受け入れ条件**:
- [ ] ユーザーは最終プロンプトをもとに、アングル変更候補を取得できる
- [ ] 候補には「カメラをもっと引く」「上から撮る」「下から撮る」「正面から見る」「斜めから見る」「横から見る」「カメラ目線」「自然な視線外し」を含める
- [ ] ユーザーは人物ポーズ変更候補を取得できる
- [ ] 候補には「立ち姿」「歩いている」「座っている」「振り返る」「片手を上げる」「バッグや小物を持つ」「服のシルエットが分かるポーズ」を含める
- [ ] 選択した候補は、元のコーディネート再現意図と人物固定プロンプトを保ったまま、再生成用プロンプトへ反映される
- [ ] 候補はユーザーが編集できる

**優先度**: P1（重要）

---

#### 撮影スタイル候補

**ユーザーストーリー**:
ユーザーとして、撮影スタイルの候補から選んで再生成用プロンプトを作りたい。なぜなら、同じコーディネートでも写真の見え方や用途を変えて試したいから。

**受け入れ条件**:
- [ ] ユーザーは撮影スタイルを「コーデ画像の撮影スタイルを使う」「人物元画像の撮影スタイルを使う」「候補から選ぶ」から選択できる
- [ ] 「コーデ画像の撮影スタイルを使う」を選んだ場合、抽出済みの撮影プロンプト部品を利用できる
- [ ] 「人物元画像の撮影スタイルを使う」を選んだ場合、人物元画像から抽出した撮影プロンプト部品を利用できる
- [ ] 候補には「ファッションスナップ」「ルックブック」「スタジオ撮影」「自然光ポートレート」「SNS投稿風」「EC商品写真風」「雑誌 editorial 風」を含める
- [ ] 選択した撮影スタイルは、コーディネート再現意図と人物固定プロンプトを保ったまま最終プロンプトに反映される
- [ ] ユーザーは撮影スタイル候補の文言を編集できる

**優先度**: P1（重要）

---

#### 背景指定候補

**ユーザーストーリー**:
ユーザーとして、背景の扱いを選んで再生成用プロンプトを作りたい。なぜなら、コーデ画像の背景を再現したい場合、人物元画像の背景を保ちたい場合、自由に背景を変えたい場合があるから。

**受け入れ条件**:
- [ ] ユーザーは背景の扱いを「コーデ画像の背景を使う」「人物元画像の背景を使う」「背景を指定しない」「候補から選ぶ」から選択できる
- [ ] 「コーデ画像の背景を使う」を選んだ場合、抽出済みの背景プロンプト部品を利用できる
- [ ] 「人物元画像の背景を使う」を選んだ場合、人物元画像から抽出した背景プロンプト部品を利用できる
- [ ] 背景候補には「街中」「カフェ」「室内」「スタジオ無地背景」「自然光の屋外」「夜景」「ショップ店内」「シンプルな白背景」を含める
- [ ] 背景を「指定しない」場合は、服装と人物固定を優先し、背景を強く縛らないプロンプトにできる
- [ ] 背景を「コーデ画像」または「人物元画像」から使う場合、どちらの画像を参照するかが貼り付け用出力で分かる
- [ ] ユーザーは背景候補の文言を編集できる

**優先度**: P1（重要）

### 将来的な機能 (Post-MVP)

#### 複数候補生成

同一の参照画像とプロンプトから、複数の画像候補を一括生成する。

優先度: P1（重要）

#### コーディネート抽出結果の構造化

抽出結果を「トップス」「ボトムス」「靴」「小物」「色」「素材」などの項目に分け、ユーザーが個別に編集できるようにする。

優先度: P1（重要）

#### プロンプトプリセット

固定人物プロンプト、ネガティブ抑制プロンプト、画風、構図、アスペクト比などをまとめたプリセットを保存できるようにする。

優先度: P2（できれば）

#### フィードバックによるプロンプトルーチン改善

蓄積したフィードバックを分析し、固定人物プロンプト、ネガティブ抑制プロンプト、最終プロンプト生成テンプレートの改善案を提示する。ユーザーが承認した改善案だけを設定へ反映する。

優先度: P2（できれば）

#### 生成結果画像の自動評価

ユーザーが外部UIで生成した画像をアップロードし、人物固定、ポーズ再現、背景、コーディネート再現、実写化の成否を自動評価する。MVPではユーザー入力によるフィードバックを優先する。

優先度: P3（将来検討）

#### 画像生成API連携

元画像と最終プロンプトを画像生成APIへ渡し、アプリ内で生成画像を取得・表示できるようにする。API呼び出しにはコストがかかるため、MVPでは実装しない。

優先度: P3（将来検討）

#### 履歴・比較ビュー

参照画像、抽出結果、完成プロンプト、生成画像を履歴として保存し、過去の生成結果と比較できるようにする。

優先度: P2（できれば）

#### コーデレイアウトテンプレート管理

雑誌風コラージュやコーデ説明画像を作るためのレイアウトテンプレートを読み込み、管理できるようにする。テンプレートは、画像スロット、テキストスロット、余白、見出し、注釈、配置ルールを定義する。

優先度: P3（将来検討）

#### 雑誌風コラージュ生成

複数の生成済み画像を素材として、コーデレイアウトテンプレートに沿って一枚の雑誌風画像に組み合わせる。生成済み画像の比較、スタイリング提案、SNS投稿用ビジュアルとして使える出力を目指す。

**想定要件**:
- 複数の生成済み画像を選択できる
- 選択した画像をテンプレート上の画像スロットへ割り当てられる
- 見出し、短い説明文、コーデ名、注釈をテンプレートに沿って配置できる
- テンプレートは別ファイルまたは設定として読み込める
- 出力は一枚の画像として保存・コピーできる

優先度: P3（将来検討）

#### コーデ説明画像生成

生成済み画像または複数の派生画像を素材として、全身、コーデのみ正面、後ろ、横などの視点別画像を組み合わせたコーデ説明画像を生成する。服装の構成や見え方を説明しやすい一枚画像を作る。

**想定要件**:
- 全身画像、正面画像、後ろ画像、横画像などをテンプレート上のスロットへ割り当てられる
- 1枚の生成済み画像から、必要な視点別画像を追加生成するためのプロンプトを作成できる
- 各視点で同じコーディネートと人物固定プロンプトを維持する
- コーデ説明用の見出し、部位ラベル、服装説明文を配置できる
- テンプレートは別ファイルまたは設定として読み込める

優先度: P3（将来検討）

## 非機能要件

### パフォーマンス

- 入力投入後、コーディネート抽出結果を 30 秒以内に返すことを目標とする
- MVPでは画像生成APIを呼び出さないため、外部APIコストを発生させない

### ユーザビリティ

- コーディネート入力、抽出結果確認、プロンプト確認、コピー出力の流れが迷わず実行できる
- 生成前に完成プロンプトを確認・編集できる
- 固定人物プロンプトとネガティブ抑制プロンプトは再利用しやすい形で保存できる
- ChatGPTへ貼り付ける際に、元画像と最終プロンプトをどのように使うかが明確である
- フィードバック入力は、問題カテゴリを選ぶだけでも次回改善につながる
- アングル・ポーズ候補は、専門的な撮影用語を知らないユーザーでも選べる表現にする
- コラージュやコーデ説明画像では、テンプレート上のどのスロットにどの画像を配置するかが分かりやすい

### 保守性

- コーディネート抽出、プロンプト合成、貼り付け用出力を独立した責務として分離する
- フィードバック記録とプロンプト改善案生成を独立した責務として分離する
- アングル・ポーズ候補はデータとして管理し、後から追加・調整できるようにする
- 撮影スタイル候補と背景候補はデータとして管理し、後から追加・調整できるようにする
- コーデ画像や人物元画像から抽出した背景・撮影・ポーズプロンプトは、コーデ抽出プロンプトとは別の部品として管理する
- コーデレイアウトテンプレートは画像生成プロンプトとは別のレイアウト定義として管理する
- 画像生成モデルやAPIを差し替えられる設計にする
- プロンプトテンプレートはコードに埋め込みすぎず、設定やデータとして管理できるようにする

### セーフティ・権利配慮

- 実在人物や第三者の画像を扱う場合は、利用権限・同意・肖像権に配慮する
- 固定人物プロンプトは、ユーザーが利用権限を持つキャラクター・人物設定を前提とする
- 生成プロンプトには、意図しない露出、年齢誤認、権利侵害、攻撃的表現などを抑制する仕組みを検討する

## スコープ外

- MVPでは高度な画像編集機能、手動マスク編集、細かなポーズ制御は扱わない
- MVP 0では固定人物プロンプト管理、ネガティブ抑制プロンプト管理、最終プロンプト合成、ChatGPT貼り付け用出力、生成結果フィードバックは扱わない
- MVP 1ではLLMによる人物固定プロンプト・ネガティブ抑制プロンプトの提案や改善は扱わない
- MVP 0〜3では生成結果画像の自動評価は扱わず、ユーザー入力によるフィードバックを優先する
- MVPでは画像生成API連携とアプリ内画像生成は扱わない
- MVPではチーム共有・権限管理・課金管理は扱わない
- MVPでは生成モデルの独自学習やファインチューニングは扱わない
- MVPでは実在人物の本人性を保証する顔再現・本人認証用途は扱わない

## 要確認事項

- 正式なプロダクト名
- 最初に貼り付け先として想定するChatGPTの使い方・画面
- 最終出力の形式（ポジティブ/ネガティブ分離、単一テキスト、手順付きテキストなど）
- 固定人物プロンプトを「テキストのみ」で管理するか、「参照画像」も含めるか
- 生成結果や履歴をローカル保存するか、クラウド保存するか
- MVP 2のLLM改善提案を、毎回の都度提案にするか、保存済みテンプレートの改善提案に寄せるか
- MVP 3のアングル・ポーズ・撮影スタイル・背景候補を、固定プリセット中心にするか、LLMによる都度提案も含めるか
